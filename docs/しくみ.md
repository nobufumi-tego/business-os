# しくみ（種明かし）

このスターターがどう作られているか。
動かしてみて「どうやってこんなこと出来るんだ？」と思った人向け。

## 二層構造：core ÷ usecases

```
[共通構造] core/      ← どの業務でも変わらない型
[ドメイン知識] usecases/  ← 業務固有の中身
```

業務改善は **構造** と **ドメイン知識** に分解できる。
構造は配れる。ドメイン知識はその人にしか書けない。
だから「**構造を配る人**」になる、という設計。

## コードではなく "指示書" で実装する

```
従来:    if is_image_pdf(): ocr_with_tesseract()  ← Pythonで分岐
このOS:  「画像PDFが来たらこう判断する」と Markdown に書く  ← Claude が読む
```

判断ロジックを `usecases/<name>/definition.md` に**日本語で書く**。
Claude がそれを読んで対話を進める。

つまり：
- 業務をわかっている人が、AIへの指示書を書ける
- コードを書かずに振る舞いを変えられる
- カスタマイズも対話で出来る

## Git は "消えない取り消しボタン"

セクションが終わるたびに自動コミット提案。
ユーザーは「保存しますか？」に yes と答えるだけ。
裏で `git add` `git commit` が走る。

リモートには絶対に push しない（`.claude/settings.json` で禁止）。
GitHub 流出事故が原理的に起きない設計。

## 棚卸し結果は "スペック" として再利用できる

`outputs/` に保存される YAML や Markdown は、ただの記録ではなく
**構造化された業務スペック**です。

これを Claude に渡せば、追加の依頼ができます：

- 「この棚卸し YAML から、改善後フローのHTMLモックを作って」
- 「この棚卸し結果を、課長報告用のスライドにまとめて」
- 「同じ部署の他業務も、この YAML と同じ形式で棚卸しを進めて」
- 「同じ業務の半年前の YAML と比較して、何が変わったか教えて」

つまり棚卸しは "入口" で、その先の作業は対話で何でも作れます。

このスターターに HTML/React 等の Web 開発フレームワークが含まれていないのは、
**含めなくても動くから**です。出力が構造化されている限り、
下流の実装は Claude との対話で十分賄えます。

## マーケットプレイスでの配布

このプラグインは GitHub の `.claude-plugin/marketplace.json` 経由で配布される。
ユーザーは `/plugin marketplace add <owner>/<repo>` で取り込むだけ。

## もっと知りたい人へ

設計の詳しい根拠は、各ユースケースの `definition.md` を読むと分かります。
そして、自社用にカスタマイズしたくなったら `/カスタマイズ` で Claude と一緒に作れます。
